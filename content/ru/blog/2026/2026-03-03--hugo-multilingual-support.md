Hugo. Поддержка многоязычности.

<!--more-->

{{< toc >}}


## <span class="section-num">1</span> Общая информация {#общая-информация}

-   Hugo имеет встроенную мощную поддержку многоязычных сайтов.


## <span class="section-num">2</span> Базовая настройка многоязычности {#базовая-настройка-многоязычности}

-   В Hugo многоязычность настраивается в конфигурационном файле.
-   Определяются языки, их параметры (например, `languageCode`, `weight` для порядка, `contentDir` для переопределения папки контента).
-   По умолчанию Hugo ожидает, что файлы контента для разных языков будут лежать в отдельных подпапках внутри `content/`, либо иметь специальные метки в именах файлов.


## <span class="section-num">3</span> Многоязычность при использовании отдельных файлов {#многоязычность-при-использовании-отдельных-файлов}

-   При подходе с отдельными файлами есть несколько способов организовать переводы.
-   Во всех вариантах изображения и другие файлы, используемые в переводах, обычно размещаются в папке `static/`.
-   Если для разных языков нужны разные версии одного изображения, придётся создавать отдельные папки внутри `static/` (например, `static/images/en/photo.jpg` и `static/images/ru/photo.jpg`) и вручную прописывать пути в Markdown.


### <span class="section-num">3.1</span> Раздельные подпапки для каждого языка {#раздельные-подпапки-для-каждого-языка}

-   Структура:
    ```shell
    content/
      en/
        about.md
        blog/
          first-post.md
      ru/
        about.md
        blog/
          first-post.md
    ```


#### <span class="section-num">3.1.1</span> Плюсы {#плюсы}

-   Чёткое разделение, каждый язык в своей папке.
-   Легко найти все страницы для конкретного языка.


#### <span class="section-num">3.1.2</span> Минусы {#минусы}

-   Нет прямой связи между переводами (Hugo связывает их автоматически по имени файла и пути, если они совпадают).
-   Ресурсы (изображения) всё ещё нужно хранить глобально в `static` или `assets`, что может привести к дублированию.


### <span class="section-num">3.2</span> Один каталог, файлы с суффиксом языка {#один-каталог-файлы-с-суффиксом-языка}

-   Можно помещать все файлы в общую папку и использовать суффиксы в именах:
    ```shell
    content/
      about.en.md
      about.ru.md
      blog/
        first-post.en.md
        first-post.ru.md
    ```


#### <span class="section-num">3.2.1</span> Плюсы {#плюсы}

-   Компактно, файлы переводов лежат рядом.


#### <span class="section-num">3.2.2</span> Минусы {#минусы}

-   При большом количестве языков каталог может стать захламлённым.
-   Ресурсы (изображения) глобальны.
-   Нет удобного способа хранить локализованные версии картинок рядом с текстом.


## <span class="section-num">4</span> Многоязычность с page bundles {#многоязычность-с-page-bundles}

-   Page bundles позволяют хранить все ресурсы страницы (текст, изображения, PDF) в одной папке, а для разных языков --- создавать параллельные папки с локализованными версиями.


### <span class="section-num">4.1</span> Структура с leaf bundles {#структура-с-leaf-bundles}

-   Допустим, у вас есть страница «О проекте» на двух языках. Вместо двух отдельных файлов вы создаёте две папки:
    ```shell
    content/
      about.en/
        index.en.md      (или просто index.md с указанием языка в front matter)
        photo.jpg        (общая картинка, если подходит для всех языков)
      about.ru/
        index.ru.md      (или index.md с языком в front matter)
        photo.jpg        (отдельная картинка для русской версии)
    ```

-   Такой подход разъединяет переводы (они в разных папках).
-   Лучший способ --- использовать одну общую папку для страницы и внутри неё размещать локализованные файлы с суффиксами:
    ```shell
    content/
      about/                     # папка страницы (bundle)
        index.en.md              # английская версия
        index.ru.md              # русская версия
        photo.jpg                # общее изображение (если одинаково для всех языков)
        diagram-ru.png           # изображение, относящееся только к русской версии
        diagram-en.png           # изображение для английской версии
        brochure.pdf             # общий PDF
    ```

-   В этом случае:
    -   Все ресурсы, общие для языков, лежат на том же уровне, что и файлы `index.*.md`.
    -   Ресурсы, специфичные для языка, также лежат рядом и могут быть использованы только в соответствующей локализованной версии.
    -   В шаблонах вы можете обращаться к ресурсам через `.Resources` и фильтровать их по языку, если задали для них параметр `lang` в front matter.


### <span class="section-num">4.2</span> Как Hugo связывает переводы {#как-hugo-связывает-переводы}

-   Внутри папки страницы файлы `index.en.md` и `index.ru.md` автоматически распознаются как переводы друг друга, потому что они находятся в одной директории и имеют одинаковое базовое имя (до суффикса языка).
-   Hugo создаёт для них страницы на разных языках и связывает их через `.Translations`.


### <span class="section-num">4.3</span> Преимущества для многоязычности {#преимущества-для-многоязычности}

-   Локализованные ресурсы.
    -   Изображения, документы и другие файлы могут быть разными для разных языков и храниться прямо в папке страницы.
    -   Например, для английской версии у вас может быть диаграмма с английскими подписями (`diagram-en.png`), а для русской --- с русскими (`diagram-ru.png`).
    -   Они лежат рядом и легко управляются.
-   Упрощённая структура.
    -   Все файлы, относящиеся к одной «логической» странице, находятся в одной папке, даже если у неё много языковых версий.
    -   Не нужно разносить по разным корневым папкам (например, `content/en/about/` и `content/ru/about/`).
-   Удобство в шаблонах.
    -   Можно использовать `.Resources.Get` для получения файла, специфичного для языка, или просто обращаться по имени, зная, что оно не конфликтует с другими языками.
    -   Например, чтобы вывести изображение для текущего языка, можно сделать `{{ $img := .Resources.Get (printf "photo-%s.jpg" .Lang) }}`.
-   Поддержка веток (branch bundles)
    -   Для разделов (например, раздел блога) используется `_index.md`.
    -   В многоязычном контексте вы можете иметь `_index.en.md` и `_index.ru.md` в одной папке раздела, а дочерние страницы организовать как подпапки с переводами внутри.
    -   Это позволяет локализовать даже структуру разделов.


### <span class="section-num">4.4</span> Пример с разделом блога {#пример-с-разделом-блога}

-   Каждый пост может иметь локализованные обложки.
-   При этом в шаблоне списка постов можно подгружать правильное изображение в зависимости от текущего языка.

<!--listend-->

```shell
content/
  blog/                      # раздел блога (branch bundle)
    _index.en.md             # заголовок раздела на английском
    _index.ru.md             # заголовок раздела на русском
    first-post/              # leaf bundle для поста
      index.en.md
      index.ru.md
      cover.jpg              # общая обложка
    second-post/
      index.en.md
      index.ru.md
      cover-ru.jpg
      cover-en.jpg
```


## <span class="section-num">5</span> Дополнительные возможности {#дополнительные-возможности}

-   Переопределение языка в front matter
    -   Можно использовать один файл `index.md` и указать язык в front matter (`lang: ru`).
    -   Но подход с суффиксами в имени файла (`index.ru.md`) часто удобнее и автоматически определяет язык без необходимости писать front matter.
-   Меню и настройки
    -   Hugo позволяет локализовать меню, параметры сайта и даже шаблоны.
-   Интернационализация строк
    -   Для интерфейсных строк (например, «Читать далее») можно использовать i18n папку с файлами переводов.


## <span class="section-num">6</span> Миграция многоязычного сайта на page bundle {#миграция-многоязычного-сайта-на-page-bundle}


### <span class="section-num">6.1</span> Конфигурация языков (`languages.yaml`) {#конфигурация-языков--languages-dot-yaml}

-   В отличие от подхода с отдельными папками (`content/ru/`, `content/en/`), здесь мы оставляем единую папку `content/` без переопределения `contentDir`.
-   Языки определяются так:
    ```yaml
    # config/_default/languages.yaml
    ru:
      languageCode: ru-RU
      languageName: Русский
      weight: 1
      params:
        description: "Описание сайта на русском"

    en:
      languageCode: en-US
      languageName: English
      weight: 2
      params:
        description: "Site description in English"
    ```

-   Не указывайте `contentDir` --- контент для всех языков будет браться из одной корневой папки `content/`.
-   Hugo сам определит язык по суффиксу в имени файла (например, `index.ru.md`) или по значению `lang` в front matter.


### <span class="section-num">6.2</span> Структура контента с Page Bundle {#2-структура-контента-с-page-bundle}

Внутри `content/` каждая страница (или раздел) представляется отдельной папкой (bundle). Внутри папки лежат файлы с языковыми суффиксами.


#### <span class="section-num">6.2.1</span> Пример для главной страницы (`_index.md`) {#пример-для-главной-страницы-_indexmd}

```text
content/
└── _index/               # папка для главной страницы (branch bundle)
    ├── _index.en.md      # английская версия
    ├── _index.ru.md      # русская версия
    └── hero.jpg          # общее изображение (если нужно)
```


#### <span class="section-num">6.2.2</span> Пример для страницы «О себе» (листовой bundle) {#пример-для-страницы-о-себе-листовой-bundle}

```text
content/
└── about/                # папка страницы
    ├── index.en.md       # английская версия
    ├── index.ru.md       # русская версия
    └── portrait.jpg      # портрет (общий или можно разные версии)
```


#### <span class="section-num">6.2.3</span> Пример для раздела блога и отдельного поста {#пример-для-раздела-блога-и-отдельного-поста}

```text
content/
├── posts/                # раздел блога (branch bundle)
│   ├── _index.en.md      # заголовок раздела на английском
│   ├── _index.ru.md      # заголовок раздела на русском
│   └── my-first-post/    # отдельный пост (leaf bundle)
│       ├── index.en.md
│       ├── index.ru.md
│       └── cover.jpg
```


### <span class="section-num">6.3</span> Как Hugo связывает переводы {#3-как-hugo-связывает-переводы}

Когда в одной папке лежат файлы с одинаковым базовым именем и разными языковыми суффиксами (например, `index.en.md` и `index.ru.md`), Hugo автоматически:

-   Создаёт две отдельные страницы с правильным языком.
-   Связывает их как переводы друг друга (доступно через `.Translations` на странице).
-   URL страниц формируются на основе пути папки и префикса языка (если включено в настройках).

В HugoBlox это работает без дополнительных действий.


### <span class="section-num">6.4</span> Настройка URL и префиксов языков {#4-настройка-url-и-префиксов-языков}

Чтобы русская версия была доступна по `/`, а английская по `/en/` (или наоборот), в конфигурации Hugo (`config/_default/config.yaml`) нужно включить префиксы:

```yaml
defaultContentLanguage: ru
defaultContentLanguageInSubdir: false  # если хотите, чтобы русский был в корне
languages:
  ru:
    weight: 1
    languageName: Русский
  en:
    weight: 2
    languageName: English
    # для английского добавить префикс /en/
    permalink: /en/:filename/
```

Либо можно управлять этим через `languages.yaml`, добавив параметр `permalink` для каждого языка. В HugoBlox часто используется встроенный переключатель языков, который учитывает эти префиксы.


### <span class="section-num">6.5</span> Перевод интерфейсных строк (i18n) {#5-перевод-интерфейсных-строк-i18n}

Текст, встроенный в шаблоны (например, «Читать далее», «Автор»), переводится через файлы в папке `i18n/`. Для каждого языка создаётся YAML-файл с переводами.

```text
i18n/
├── ru.yaml
└── en.yaml
```

Пример `ru.yaml`:

```yaml
"Read more": "Читать далее"
"Next": "Следующий"
"Previous": "Предыдущий"
```

Пример `en.yaml`:

```yaml
"Read more": "Read more"
"Next": "Next"
"Previous": "Previous"
```

Эти строки будут автоматически подставлены в шаблоны HugoBlox в зависимости от текущего языка страницы.


### <span class="section-num">6.6</span> Особенности HugoBlox: виджеты и параметры {#6-особенности-hugoblox-виджеты-и-параметры}

В HugoBlox многие элементы настраиваются через виджеты на главной странице (например, `about`, `experience`). Для их локализации есть несколько путей:

-   **Использовать отдельные файлы контента для каждого языка**, как показано выше. Например, виджет «about» может быть реализован через страницу `content/about/index.en.md` и `index.ru.md`. Внутри этих файлов вы определяете параметры виджета (через front matter) на нужном языке.

-   **Локализация параметров виджета** возможна через `languages.yaml`. Например, если виджет использует параметр `bio` из конфига, вы можете задать разные значения для каждого языка:

<!--listend-->

```yaml
# config/_default/languages.yaml
ru:
  params:
    author:
      bio: "Биография на русском"
en:
  params:
    author:
      bio: "Biography in English"
```

В шаблоне виджета тогда используется `.Site.Language.Params.author.bio`.

-   **Для меню** можно создать отдельные файлы меню для каждого языка: `config/_default/menu.ru.yaml` и `config/_default/menu.en.yaml`. Это позволит задать разные пункты и ссылки.


### <span class="section-num">6.7</span> Важные замечания {#7-важные-замечания}

-   **Имена файлов**: Для листовых страниц используйте `index.LANG.md`, для разделов --- `_index.LANG.md`. Если язык основной (например, русский) и вы хотите, чтобы файл без суффикса тоже считался русским, укажите в его front matter `lang: ru`. Но проще использовать суффиксы для всех языков --- это явно и надёжно.
-   **Ресурсы внутри bundle**: Изображения и другие файлы, общие для всех языков, кладите прямо в папку страницы. Если нужны разные версии для разных языков, можно добавить суффикс языка в имя файла (например, `diagram-ru.png`, `diagram-en.png`) и в шаблоне выбирать нужный через `.Resources.GetMatch` с учётом `.Lang`.
-   **Проверка связей**: После настройки запустите `hugo server` и откройте сайт. На странице должна быть возможность переключения языка (если тема поддерживает). Также можно вывести в шаблоне список переводов: `{{ if .IsTranslated }}<select>...{{ end }}`.
-   **Смешанная структура**: Если у вас уже есть старый контент в отдельных папках (`content/ru/`, `content/en/`), вы можете постепенно мигрировать страницы в новый формат, удаляя старые файлы. Главное --- не оставлять дублирующихся путей, чтобы не возникало конфликтов.


### <span class="section-num">6.8</span> Пример готовой структуры {#8-пример-готовой-структуры}

```text
content/
├── _index/                     # главная
│   ├── _index.en.md
│   └── _index.ru.md
├── about/                      # страница "О себе"
│   ├── index.en.md
│   ├── index.ru.md
│   └── avatar.jpg
├── posts/                       # раздел блога
│   ├── _index.en.md
│   ├── _index.ru.md
│   └── my-first-post/
│       ├── index.en.md
│       ├── index.ru.md
│       └── cover.jpg
├── projects/                    # раздел проектов
│   ├── _index.en.md
│   ├── _index.ru.md
│   └── project1/
│       ├── index.en.md
│       ├── index.ru.md
│       └── screenshot.png
└── contact/                     # страница контактов
    ├── index.en.md
    └── index.ru.md
```


### <span class="section-num">6.9</span> Заключение {#9-заключение}

Использование page bundle с локализованными файлами внутри одной папки --- это чистый и удобный способ организации многоязычного контента в Hugo/HugoBlox. Он обеспечивает локальность ресурсов, автоматическую связь переводов и простоту управления. Все настройки HugoBlox (виджеты, меню, i18n) прекрасно с этим работают, если правильно организовать файлы контента и параметры в конфигурации.

Если вы мигрируете существующий сайт с отдельными языковыми папками, действуйте постепенно: для каждой логической страницы создавайте новую папку, переносите туда файлы с языковыми суффиксами, обновляйте ссылки на изображения и удаляйте старые файлы. После завершения миграции не забудьте убрать `contentDir` из `languages.yaml`, чтобы Hugo не искал контент в старых местах.
